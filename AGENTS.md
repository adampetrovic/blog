# AGENTS.md

This repository is Adam Petrovic's personal Hugo site, deployed to GitHub Pages.

## Project shape

- Static site generator: Hugo.
- Theme: `hugo-bearblog` as a Git submodule in `themes/hugo-bearblog`.
- Site config: `hugo.toml`.
- Source content: `content/`.
- Generated output: `public/`.
- Deployment workflow: `.github/workflows/pages.yaml`.

Do source edits in `content/`, `hugo.toml`, `layouts/`, or `static/`. Do not hand-edit `public/`; it is generated by Hugo.

## Local commands

Run these from the repository root:

```bash
hugo server -D
hugo --gc --minify
```

Use `hugo server -D` for local preview, including draft content. Use `hugo --gc --minify` before finishing changes to verify the production build.

## Creating blog posts

Blog posts live under `content/blog/`. Create one Markdown file per post:

```text
content/blog/my-post.md
```

Recommended front matter:

```toml
+++
title = "My Post Title"
date = 2026-05-27T00:00:00+10:00
draft = false
slug = "my-post"
tags = ["hugo", "homelab"]
+++
```

Notes:

- The `blog` section has a permalink override in `hugo.toml`, so posts publish at `/:slug/` rather than `/blog/:slug/`.
- The blog listing is at `/blog/` via `content/blog/_index.md`.
- Tags are enabled via `tags = ["tag-name"]` and generate tag pages under `/blog/tags/<tag>/`.
- Use `<!--more-->` to control summary/excerpt cut-off where useful.
- Use Sydney/Australia dates and timezone offsets when adding dated content.
- Keep slugs stable once published; changing them breaks existing URLs unless redirects/aliases are added.

## Creating regular pages

Top-level standalone pages live directly under `content/`:

```text
content/about.md
content/projects.md
```

Recommended front matter:

```toml
+++
title = "Projects"
menu = "main"
weight = 20
+++
```

A page without `menu = "main"` will still be generated, but will not appear in the header navigation unless the theme or another page links to it.

## Navigation links

There are two common ways to add nav items.

### Add a content page to the nav

Put this in the page front matter:

```toml
menu = "main"
weight = 20
```

Lower weights appear earlier. Current convention:

- Home: `weight = 10`
- Other internal pages: choose weights between existing entries.
- Blog listing: configured in `content/blog/_index.md`.

### Add a config-only nav link

Use `[[menus.main]]` in `hugo.toml` for links that do not correspond to a content page, such as PDFs or external links:

```toml
[[menus.main]]
  name = "Example"
  url = "https://example.com/"
  weight = 30
```

Prefer page front matter for normal site pages. Use config-only menu entries sparingly.

## Static assets

Files in `static/` are copied to the site root. For example:

```text
static/resume.pdf -> /resume.pdf
static/CNAME      -> /CNAME
```

Use `static/` for PDFs, downloads, favicons, and other files that should be served as-is.

## RSS

The custom RSS template is `layouts/_default/rss.xml`. It currently includes regular pages from the `blog` section only:

```go-html-template
{{- $pages := where .Site.RegularPages "Section" "blog" -}}
```

If the section name changes, update this template as well as `hugo.toml` permalinks and content directories.

## Deployment lifecycle

1. Make source changes locally.
2. Run `hugo --gc --minify` to verify the site builds.
3. Commit and push to `main`.
4. GitHub Actions runs `.github/workflows/pages.yaml` on every push to `main`.
5. The workflow checks out submodules, installs Hugo Extended, builds with:

   ```bash
   hugo --gc --minify --baseURL "${{ steps.pages.outputs.base_url }}/"
   ```

6. The generated `public/` directory is uploaded as a Pages artifact.
7. GitHub Pages deploys the artifact.

The repository has a generated `public/` directory, but deployment rebuilds from source. Treat source files as canonical.

## Typesetting and formatting capabilities

The theme and Hugo/Goldmark setup support rich Markdown with syntax highlighting. `hugo.toml` enables unsafe Goldmark rendering, so inline HTML is allowed when useful.

Supported patterns include:

### Headings

Use standard Markdown headings:

```markdown
# Heading level 1
## Heading level 2
### Heading level 3
#### Heading level 4
```

Prefer one H1-equivalent title from front matter, then start body sections at `##`.

### Inline text

```markdown
A paragraph with **bold**, _italic_, **_bold italic_**, `inline code`, and [links](https://example.com/).
```

Hard line breaks work with two trailing spaces, but prefer paragraphs unless a line break is intentional.

Smart punctuation is enabled by Hugo/Goldmark, so quotes, dashes, and ellipses render nicely from normal Markdown text.

### Lists

Unordered, ordered, nested, and task lists are supported:

```markdown
- Item
  - Nested item

1. First
2. Second

- [x] Done
- [ ] Todo
```

### Blockquotes

```markdown
> Good systems are boring in production.
>
> The interesting work happens before they get there.
```

### Code blocks

Use fenced code blocks with language identifiers for highlighting:

````markdown
```go
package main

func main() {}
```

```yaml
apiVersion: apps/v1
kind: Deployment
```
````

Highlighting is configured in `hugo.toml` with the `friendly` style and line numbers enabled.

### Tables

```markdown
| Area | Tool | Notes |
| --- | --- | --- |
| Infra | Talos | Immutable Kubernetes nodes |
```

### Horizontal rules

```markdown
---
```

Use sparingly to separate major sections.

### Images

```markdown
![Alt text](https://placehold.co/800x300?text=Example)
```

Prefer meaningful alt text. Local images can be served from `static/` and linked as `/image-name.png`.

### Footnotes

```markdown
This sentence has a footnote.[^1]

[^1]: Footnotes are useful for side notes.
```

### Definition lists

```markdown
Kubernetes
: A container orchestration system.
```

### HTML details blocks

Because unsafe Goldmark rendering is enabled, expandable details blocks work:

```html
<details>
<summary>Click to expand</summary>

Hidden content can include Markdown-style emphasis and plain HTML.

</details>
```

### Symbols and technical prose

Unicode symbols render directly and are suitable for technical writing:

```markdown
latency ≤ 100ms, availability ≥ 99.9%, and p95 → p99
```

## Style guidance

- Keep the site small, plain, and fast.
- Prefer clear Markdown over custom HTML.
- Use front matter titles in title case.
- Use stable slugs for published pages.
- Run the Hugo production build before committing.
